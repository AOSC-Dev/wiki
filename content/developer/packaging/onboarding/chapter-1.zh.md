# 第一章 走进安同

安同，意为 “安于同学合作”，是一个独立的社区，基于志愿、兴趣与合作存在。在这里您可以认识到各位业界大佬，与他们一同维护安同 OS 及周边项目。社区以贡献者的发展为运作目标，社区内的项目均以研究、学习和教育为目的开发，社区也希望贡献者在参与开发的过程中能够积累相关经验、提升自身技能、学习新的知识。

安同 OS 是安同开源社区的主要项目，同时也是茫茫大海之中的一个 Linux 发行版。您也许已经知道，安同 OS 以 “塞爆” 闻名——不拆分系统依赖、一个软件包解决一切，且使用 Debian 的包管理套件 dpkg 和 APT。与此同时，基于 `libapt-pkg` 的[小熊猫包管理器 (oma) ][oma]具有比 APT 更友好的设计和体验，因此正在逐渐取代 `apt` 命令。

# 社区文化

在您加入安同开源社区之前，您需要阅读并了解[《安同开源社区人际关系准则（长期意见征集稿）》][aosc-ipg]。这是安同社区内交流与合作的基本原则，敬请遵守。

在当今 AI 辅助编程盛行的当下，我们要求您不能指示您的 AI 工具完成安同开源社区内的任何贡献。您可以参考 AI 工具生成的代码、文档等内容，但您必须人工参与任何编写、测试及发布的过程。您可以阅读[《安同开源社区 AI 工具使用及披露约定》][aosc-ai-tool-guidelines]，了解安同开源社区的 AI 工具使用政策。

## 成员组成

安同开源社区由数十个来自世界各地的开发者们共同参与，维护社区内的项目。

社区不分组织架构——您在日常中听到的 “特首”（指[白铭骢][mbai-github]）只是口口相传的称呼而已；尽管看上去决策基本由特首进行，但特首也正在将决策工作下放给各位贡献者。社区成员都是贡献者，不论资历深浅、年龄大小、能力强弱，都应该对各种事情表达自己的意见。

## 社区的工作

您和大家作为志愿付出时间的贡献者，社区无法、也不能要求您付出时间。您可以在任意时间选择任意工作，但是您应该意识到社区的工作从来都不是一个人进行的，所以请在主群及贡献者群组内保持沟通。作为（潜在或已经加入的）贡献者，您可以做的工作主要分以下几种：

- 技术工作：
    - 安同 OS 维护：引入、更新及修缮软件包，协助更新重要组件、协助实现新的系统特性等
    - 安同 OS 基础设施（如构建工具链）维护：配合实现新系统特性、修缮代码、实现新打包功能等
    - 安同 OS 周边项目（小熊猫包管理器、系统发行及安装镜像生成器、包管理基础设施 `p-vector` 等）的维护
- 文书工作：
    - 编写、翻译及完善文档（安同 OS 及周边项目的使用指南、构建及包管理基础设施的技术参考文档等；也包括这篇文档）
    - 编写社区期刊《安记冰室》、特辑《聊斋》；具体内容请见[宣发内容说明][newsroom-readme]
    - 编写及完善网页的文案
- 日常维护：
    - 协助测试 PR 内的软件包，在 PR 内提交测试结果
    - 审阅 PR 的代码，给出审阅建议
    - 协助测试未完成的周边项目等
- 人际交流：
    - 协调贡献者之间出现的分歧
    - 协助调查用户报告的问题
    - 协助与愿意合作的上游厂商交流
- 外宣工作：
    - 协助维护社区各宣发平台（微信公众号、微博、哔哩哔哩、知乎、X (Twitter) 等）
    - 协助维护社区门户

本入门指南主要介绍安同 OS 的维护工作。

## 贡献者及社区内交流

成为贡献者后，您需要加入 AOSC 贡献者群组。贡献者群组主要讨论新的想法、社区内部事务及需要征求贡献者意见的事务。

在[社区附属聊天群组、论坛][aosc-contact]讨论交流时，请尽量保持文明，并注意以下几点：

- 礼貌用语，拒绝攻击。对方再怎么烦人，他们依旧是人。您可以拒绝回应，但您不能任性地言语攻击对方。
- 放下架子，心平气和。大家出于兴趣爱好、志愿付出时间维持社区工作。获得认可不等于高人一等，否则您的态度势必招来贡献者的厌恶。
- 尽量忍耐，听取意见。合作是社区的根本，无论意见尖锐与否，您都不应该在贡献者内挑起争端，这是毫无建设性的。耐心地听取意见有助于您的项目继续推进。
- 遇到问题，不懂就问。尽管您可以推进任何更改，但社区依旧需要坚持一些底线。您在遇到问题、需要做出一些改动时，请先征求贡献者的回复。拿捏不定时，也请不要擅自做主。
- 正视用户，不要敷衍。不论是回答用户的问题、还是提供建议，复制粘贴 LLM 输出是不负责任且不尊重对方的行为。

# 维护体系

尽管安同 OS 使用 Debian 的包管理体系，但在源码树方面却是独立的——安同 OS 不使用 debhelper 打包，因此软件包的源码也不是基于 debhelper 编写和构建的。安同 OS 有一套自己的开发环境和软件包维护体系。其中包括如下开发工具链：

- 源码构建系统 Autobuild：负责读取软件包定义、根据脚本编译源码，最终打包出软件包。
- 源码管理系统 ACBS (Autobuild CI Build System)：负责读取源码树，下载、解压源码，然后调用 Autobuild。
- 构建环境容器管理器 Ciel：负责管理运行 ACBS 和 Autobuild 的环境。
- ABBS 源码树：这里存放所有软件包的源码信息、软件包元数据、构建脚本和下游补丁。

此外，社区的自动化构建设施也较为完善。成为贡献者后，您可以轻松地通过社区的自动化设施维护软件包。这些设施包括：

- 自动打包机器人 BuildIt!：您只需在聊天框中输入命令，构建作业将由机器人自动发起。集成了安同 OS 的开发工作流。
- 自动更新检查设施 AutoPR: 定期检查软件包的更新，发起 PR 然后自动执行构建。
- 更新追踪设施 Anicca：定期检查软件包更新，以便 AutoPR 自动发起更新作业。

安同 OS 迭代的方式较为特别，我们称之为 “主题制” 迭代方式。主题制迭代方式是安同 OS 处于 “半滚动” 迭代状态的主要原因。

# 了解测试源 (Topic)

安同 OS 遵循一套基于测试源的维护机制。测试源（Topic，也叫 “主题”），是较其他发行版比较独特的概念。安同 OS 更新软件包的方式既不像 Debian 那么啰嗦，也不像 Arch Linux 那样直接。笼统地说，测试源是一次安同 OS 软件包更新或变动：更新软件包，开一个测试源；修软件包的 Bug，开一个测试源；一系列组件更新，开一个测试源。

测试源如何开展取决于您改动（更新）软件包的目的，因此一个测试源的修改量可以是一个、多个甚至很多个包。一些例子如下：

- Htop 发布新版，开一个 `htop-x.y.z` 测试源更新 Htop；由于应用类软件没有反向依赖，此类测试源的修改最直接
- `libxml` 发布新版，但由于 `libxml2` 是共享库，且新版通常会更新其 API 版本，因此需要发起 `libxml2-survey-YYYYMMDD` 测试源，带上其所有反向依赖
- KDE 发布 KDE 6，发起 `kde-6` 测试源时需要带上整个 KDE Plasma 套件，以及整个 KDE 应用程序套件内的软件包
  - 此类更新需要长期测试，因此该测试源需在达到合并质量前始终保持维护状态

测试源在 ABBS 中的体现是 ABBS 源码树中的分支。除 `stable` 分支之外，其余的分支均为测试源分支。同样地，测试源在安同 OS 中的体现方式是 `oma topics` 命令中列出的测试源列表，其中列有测试源的描述（对应分支对应的拉取请求标题）及测试源名称（对应 ABBS 树的分支名）。在软件源中，测试源即为 `dists/` 和 `pool/` 目录中的发行（相当于 Debian 中的 stable, unstable, oldstable）。

所有更新和改动均通过拉取请求 (PR) 进行，拉取请求合并前，感兴趣的用户或开发者将有机会测试测试源内的软件包，尽可能地避免合并后出问题的情况。测试源分支在构建、审阅并测试通过后，会合并到稳定源（ABBS 树中的 `stable` 分支），以此将改动推送给所有用户。测试源合并前可供所有人加入，在 Topic 的生命周期内提供测试结果及反馈。

# 安同 OS 的开发工具链

以下是安同 OS 开发中需要的工具，这些工具在安同 OS 维护过程中相辅相成、各司其职。因此我们也可以称这组工具为 “工具链”，以体现这些工具间相互承接的关系。

## Autobuild4

Autobuild4 处于安同 OS 开发环节中最底层的一环。Autobuild4 负责读取软件包定义及构建脚本（即 `autobuild/` 目录下的内容），根据定义文件配置、编译并安装软件，最终打出软件包。然而 Autobuild4 并不负责软件包源码的下载和管理，这些工作由 ACBS 负责。

Autobuild4 的构建脚本、软件包定义均为 Bash 脚本，其中软件包定义严格限制为安同 OS 软件包描述语言（APML，为 Bash 的子集）。

## ACBS

ACBS（全称 Autobuild CI Build System），处于 Autobuild 之上，是用于管理安同 OS 软件包源码的工具。ACBS 负责读取软件包版本及源码信息（即 `spec` 文件），下载、解压源码，而后调用 Autobuild4 打包。

安同 OS 的所有软件包源码定义及构建配置都存放在 [ABBS 树][aosc-os-abbs]中。由于 ACBS 会自动根据软件包安装情况分析构建序列，因此，执行一次 ACBS 可能会根据推导出的构建序列，调用 Autobuild4 构建出多个软件包。

## Ciel

虽然 ACBS 能够推导构建序列，能够一次性构建多个软件包，但 ACBS 不会保证系统环境的清洁（如是否包含未合并的的修改、系统内是否安装有隐性依赖等）。在同一个环境内构建所有软件包，会导致构建出的软件带上了构建时没有显式指定的依赖。因此构建工作流中需要标准的构建环境，配合可以随时回滚的容器，保证构建环境不会带入没有指定的依赖。

Ciel 处在开发工具链的最上层，是安同 OS 的构建容器管理器。Ciel 负责更新、回滚等容器管理工作，并调用 ACBS 构建软件包。Ciel 使用 OverlayFS，以便在每次构建之间回滚容器，保持干净的构建环境。由于安同 OS 深度使用 Systemd，因此 Ciel 使用 systemd-nspawn 管理构建容器。

Ciel 负责用容器机制管理构建环境，并在每次构建成功后回滚容器。在容器启动后，Ciel 会自动根据命令调用 ACBS 拉取源码、发起构建。每次构建完毕后，Ciel 会收集构建出的软件包至 Ciel 工作区目录下，供开发者上传或测试。

安同 OS 开发环节中严格使用 Ciel 容器打包，确保构建环境可靠且可复现。

## BuildIt!

BuildIt! 是安同 OS 的自动化构建工具，它负责将作业派发到合适的构建机上，自动调用 Ciel 发起构建。您只需向 Telegram 机器人下达构建指令，BuildIt 会自动为您构建，并将构建结果反馈到拉取请求中。因此，它深度集成了安同 OS 的开发工作流。但是，BuildIt! 只有在您成为贡献者后才能使用，敬请注意。

# 总结

- 安同开源社区内的工作不仅仅是维护安同 OS 本身，还有许多周边项目需要开发及维护，以及美工、文书及宣发工作
- 安同 OS 以测试源的形式更新及维护软件包，在未达到合并质量前不会合并至主线
- 安同 OS 的开发工具链包含 Autobuild4（打包工具）、ACBS（源码处理工具）和 Ciel（容器管理器），以及一套标准的打包环境 (BuildKit)
- 安同 OS 的打包脚本及下游补丁所在 Git 仓库为 [ABBS][aosc-os-abbs]，其中除 `stable` 分支外均为测试源分支
