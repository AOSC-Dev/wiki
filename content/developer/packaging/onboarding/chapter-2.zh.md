在开源世界，任何项目在发版后基本只分发该版本的源码包（俗称 Tarball，以 `tar` 格式归档的源码得名）。任何感兴趣的用户需要自行将其编译并安装到自己的系统中才能使用新发版的软件。发行版统一了编译平台，并直接以软件包的形式分发编译好的二进制软件，方便用户安装和使用。本节内容将带您探索发行版打包时使用的编译器及用法，以及一些行业黑话。

# 编译器和工具链

**“编译 (Compile)”** 一般指将文本形式的源码经多次处理及转换后产生可执行的二进制、可加载的动态链接库（或称共享库, Shared Library）的过程。**编译器 (Compiler)** 就是用于执行编译过程的程序。

现代编译器分为前端（读取源码，分析并产生中间文件）和后端（将中间文件转换为目标架构的汇编指令）两部分，以此支持多种编程语言和架构。因此，现代编译器实际上可以称为**编译器套件**。

不过严格地讲，从源码到二进制文件需要经过多个步骤，这些步骤由专门的程序负责：

1. 编译器 (`gcc`) 读取源码文件，生成汇编文件
2. 汇编器 (`as`) 将编译器生成的汇编文件汇编成二进制输出文件（此时该文件还无法执行，为可重定位文件）
3. 链接器 (`ld`) 将一系列可重定位二进制组合起来，修缮函数地址及指定的动态链接库的符号信息，链接成可执行二进制或动态链接库

{% card(type="tips") %}
由于用户执行的 “编译器” 本身只是用于驱动编译器后端的程序，因此默认情况下 “编译器” 会自动完成整套步骤。
{% end %}

尽管上述是一个简化的过程，但也可以看出，从源码到二进制的可执行程序的过程中，下一个环节要使用前一个环节生成的内容，形成了一个链条。我们把编译器和用来最终生成可执行文件的程序的组合称为工具链。

此外，C 和 C++ 语言编写的程序还需链接发行版统一使用的 C 运行时库（提供 C 语言标准库实现）和 C++ 运行时库（提供 C++ 语言标准库实现）。不同 C 库的链接库文件名不同，其实现的函数及对应的符号也不同，因此发行版提供的编译器固定面向同一套 C 和 C++ 运行时库实现。主流的 C 标准库实现有 GNU glibc（为绝大多数发行版所使用）、musl（轻量级实现）和 Bionic（为 Android 系统所使用）。

一般情况下，绝大多数发行版的工具链中，编译器由 GCC 提供，汇编器及链接器由 GNU Binutils 提供，C 运行时库由 GNU glibc 提供，C++ 运行时库由 GCC 随附的 `libstdc++` 提供。

# 发行版维护相关考虑

开始做发行版之前，我们一般考虑以下方面：

1. 架构支持范围：只支持 x86-64 架构，还是维护多个架构的移植？
2. 编译器支持：GCC 和 Binutils 若不支持该架构，则该移植将无法开始（无法构建任何代码）
3. Linux 内核支持：Linux 中若不支持该架构，该移植将无法进行（没有可以运行的内核）
4. GNU glibc 支持：如果 glibc 中没有该架构支持，则无法对该架构维护发行版相关组件（缺少 C 运行时）

{% card(type="info", title="为什么选择 glibc") %}
绝大多数发行版选择 GNU glibc 的原因主要有：

- 历史原因：Linux 内核最初发布时 GNU 项目已经几近完整，只差其 Hurd 内核没有完成开发。GNU 项目于是计划利用 Linux 内核加上 GNU 已经完备的用户组件，组成完整的 GNU 操作系统生态，因此便有了 “GNU/Linux” 这个名字
- 性能：glibc 针对所有其支持的架构均有非常激进的优化（如，手动编写的、用汇编语言实现的特定函数），保证链接 glibc 的程序拥有最佳性能
- 特性完整：glibc 不仅性能出色，还拥有 GNU 自己的特性扩展，以增强特定情况下的功能，如 Unicode 支持、国际化支持等
- 用途广泛：长期以来，绝大多数 Linux 发行版均使用 GNU glibc 作为其使用的 C 运行时库
{% end %}

此外，越来越多发行版所依赖的软件项目开始迁移至更 “安全” 的 [Rust 编程语言][rust-lang]，但由于 Rust 使用的 LLVM 后端支持范围较窄，导致发行版无法为一部分 Linux 内核仍旧支持的架构维护完整特性集（如 DEC Alpha、PA-RISC 等）。

# 主流的编译器

绝大多数发行版均选择 [GCC][gcc] 作为编译整个系统时使用的编译器。有少部分软件项目推荐 [LLVM/Clang][llvm] 作为其编译器，理由是 LLVM 有更加先进的特性，如自动向量化等，因此在特定用途下性能较 GCC 编译的程序略高。

但是，市面上还有一些其他类型的编译器（如，`tcc`），但由于其为基于兴趣开发的项目，或其架构支持或语言标准支持不完整，不适合作为构建发行版时使用的编译器，故在此不做介绍。

## GNU 编译器套件 (GCC)

GNU 编译器套件 (GNU Compiler Collection, GCC)，是 GNU 项目发起并维护的编译器项目。GCC 支持 C、C++、Objective-C/C++、Go、Fortran、Ada、D 等语言，且其构建出的程序运行稳定，因此基本认为是行业标准。

GCC 套件内一般包含以下组件（部分组件列出了其程序名）：

- `gcc`、`gdc`、`f95`、`gnat`: 编译器驱动前端（Driver，本身不负责编译程序，而是根据参数及提供的文件调用对应语言的编译器）
- `cc1`: C 语言编译器
- `cc1plus`: C++ 语言编译器
- `cc1gm2`: Modula-2 语言编译器
- `f951`: Fortran 语言编译器
- `d21`: D 语言编译器
- `gnat1`: Ada 语言编译器
- `libgcc`: GCC 编译器运行时，提供了 GCC 内建函数及部分数学运算的软件实现
- `libstdc++`: GCC 随附的 C++ 语言标准库实现

GCC [支持 50 余种架构][gcc-config]（以 `gcc/config` 中的文件夹为准，不同架构的语言支持情况不同），但 GCC 一经编译，就只能支持固定的单个架构。因此若需要为其他架构编译程序，需要单独为目标架构编译 GCC 及 Binutils。

## LLVM

[LLVM][llvm] 是 LLVM Project 发起的编译器项目。起初 LLVM 全名为 “底层虚拟机 (Low Level Virtual Macine)”，但随着 LLVM 项目的演进，该全称已失去其意义。LLVM 项目包含 LLVM 通用编译器后端，以及数个编程语言前端：

- `llc`: LLVM 中间语言 (LLVM IR) 编译器
- `lld`: LLVM 项目的链接器实现
- `clang`: C/C++ 编译器前端
- `flang`: Fortran 语言编译器前端
- `libc++`: LLVM 的 C++ 标准库实现
- `compiler-rt`: LLVM 项目的编译器运行时库实现（对应 `libgcc`）

相对于 GCC，LLVM [支持的架构较少][llvm-targets]。但 LLVM 编译器套件非常灵活，编译出的 LLVM 套件可以支持多种架构。即便如此，如需为其他架构编译程序，仍需对应架构的 Binutils 和 C 标准运行库等组件。

此外，LLVM 后端的模块化实现使得其他项目能够轻松集成 LLVM，如 Rust 编译器 `rustc` 就使用 LLVM 作为其编译器后端，因此 Rust 语言的架构支持基本与 LLVM 的架构支持强绑定。

# 运行编译器

{% card(type="tips") %}
安同 OS 采用 GCC 作为标准发行版编译器。因此除非另行说明，本入门指南均以 GCC 为准。
{% end %}

编译器的命令行设计形式一般为：

```sh
编译器程序名 [其他编译器参数] -o 输出文件 输入文件 [输入文件 2 ...]
```

例如，编译并运行 Hello World 程序：

```console
$ gcc -o hello hello.c
$ ./hello
Hello, World!
```

# 编译器参数的分类

您已经注意到上面的命令行示例中可以指定 “其他编译器参数”。虽然不指定其他任何编译器参数时也能编译出程序，但编译器参数可以更精细地控制编译过程的每个方面。

GCC 套件中定义了[数百个编译器参数][gcc-options]，这些编译器参数主要分为以下几类：

- 目标架构微调：指定生成的代码至少要在哪一代指令集的处理器上运行 (`-march`)，以及在哪一代指令集的处理器上性能最佳 (`-mtune`)，以及可能需要关闭或打开的指令集扩展等
- 代码优化：控制编译过程中代码经过的优化流程，一般编译器优化预设分四个等级，以及一些特殊的优化预设
- 代码加固：自动在生成的代码中插入加固代码，如栈溢出保护，以及使用 C 标准库随附的加固功能
- 运行时异常监测：自动在生成的代码中插入或链接监测运行时各种未定义行为的代码
- 控制链接过程：调用 `gcc` 等前端驱动程序时可以向链接器传递选项，如需要链接的动态库、动态库查找位置等
- 宏定义：除了代码中的 `#define` 之外，可以在命令行中通过 `-D` 及 `-U` 参数定义或取消定义预处理宏
- 头文件控制：指定预处理期间的头文件搜索目录
- 编译器行为控制：指定编译时遵循的语言标准 (`-std=`)、将所有编译警告视为错误 (`-Werror`) 等会影响编译器行为的参数

{% card(type="tips") %}
命令行重复指定的同类型参数中，后者的效果会覆盖前者：

```sh
gcc -O0 -O3 -O2 -march=x86-64 -march=native ...
```
相当于：
```sh
gcc -O2 -march=native ...
```
{% end %}

接下来，我们将一一讲述上述方面中较为重要的编译器参数。

# 架构微调参数

GCC 中所有架构相关的参数均以 `-m` 开头。每个架构必须自行定义架构相关的选项。这些选项在以下方面控制生成的代码：

- 架构基线：生成的代码必须至少在指定的指令集版本的处理器上运行
- 指令位宽：生成代码的位宽（32 或 64 位）
- 指令集扩展：精准控制生成代码时可以生成指定指令集扩展的指令
- 浮点运算：如果代码中含有浮点数运算，则生成硬浮点（使用硬件浮点单元）或软浮点（调用 `libgcc` 或使用软件实现）的代码
- 指令集缺陷处理：生成能够绕过特定指令集缺陷的代码
- 端序控制：生成大端序或小端序代码

其中指令位宽、浮点运算方式及端序控制选项在构建 GCC 时已被记录为默认预设，因此绝大多数情况下无需显式指定。

## 发行版的基线 (Baseline)

随着处理器的演进，其使用的指令集也会随之扩展。一般来讲，新一代指令集可以向后兼容之前版本的指令集。为旧版指令集编译的程序可以在任何之后版本指令集的处理器上运行，但反之则无法运行：为新版指令集编译的程序使用了旧版指令集不支持的指令。

基线定义了发行版分发的软件最低可以运行的指令集版本。发行版的基线通常非常保守，以保证发行版中的软件可以在较广范围的处理器上运行。

### 发行版基线定义的例外情况

尽管如此，不是所有软件项目都可以运行在发行版基线定义的最低指令集版本的处理器上。其背后的原因基本可以认为只有一个：此类软件为了加快运行速度，在代码中使用了新的指令集扩展（尤其是向量指令集扩展）处理数据，而带有该指令集扩展的指令集版本比发行版基线定义的高。

典型的例子均为模拟器、专业用途或科学用途的软件，如 Blender 在 x86-64 架构上最低要求支持 SSE4.2 指令集扩展的处理器，而 SSE4.2 不是大多数发行版定义的基线（初代 x86-64）的一部分；RPCS3（PS3 模拟器）尽管支持初代 x86-64 指令集，但在支持 AVX 和更新版本的向量指令集扩展的处理器上性能更佳；Eden（Nintendo Switch 模拟器）则要求处理器至少支持 FMA 指令集扩展。

## `-march` 和 `-mtune`

发行版的基线均通过编译软件时指定 `-march` 和 `-mtune` 参数来实现：

- `-march` 定义生成的代码面向的指令集版本
- `-mtune` 定义代码调优期间面向的微架构或核心

`-mtune` 不会生成面向调优目标指令集版本的代码，该参数会保持 `-march` 指定的指令集版本，而在指令调度、分支预测、缓存对齐等方面为指定的微架构优化，使编译的程序在 `-mtune` 指定的微架构处理器上性能最佳，但依旧可以在其他版本指令集的处理器上运行。

部分 GCC 支持的架构可能不提供 `-mtune=` 选项。`-march=` 和 `-mtune=` 的值由每一个架构支持后端自行规定。

以下列举几个例子：

- `-march=x86-64 -mtune=arrowlake`: 生成面向初代 x86-64 指令集架构的代码，但面向 Arrow Lake 在指令层面优化生成的代码
- `-march=arrowlake -mtune=arrowlake`: 生成面向 Arrow Lake 微架构支持的指令集的代码，同时面向 Arrow Lake 微架构调优；生成的程序会尽量使用 Arrow Lake 支持的指令集中的指令（例如 AVX2）
- `-march=armv9-a -mtune=cortex-a710`: 生成面向 ARMv9.0-A 指令集的代码，面向基于 Cortex-A710 核心的处理器调优（更早的核心不支持 ARMv9.0-A 指令集）

不过，在 Intel 处理器家族中有一处例外：

- AVX-512 指令集扩展只在一部分家族的处理器中支持，Intel 在其 12 代 Core 处理器中禁用了该指令集扩展的支持，并在随后更新的处理器中移除。因此如果面向带有 AVX-512 指令集支持的处理器编译程序，这些程序将无法在 12 代及之后的 Core 处理器上运行。

## 指令集扩展开关

指令集扩展开关可以更加精细地控制代码生成时采用的指令。`-march` 参数会隐含地启用一些指令集扩展开关，但也可以在 `-march` 的基础上追加或禁用指令集扩展。一旦额外追加指令集扩展，生成的程序就不能在不支持该指令集扩展的处理器上运行。因此发行版开发中不允许使用此类选项。下面列举一些常用架构的指令集扩展开关：

GCC 中指令集扩展开关遵循 `-m<指令集扩展名>` 的格式，如 `-msse2`（启用 SSE2）、`-mlsx`（启用 LASX）；关闭指令集扩展的选项遵循 `-mno-<指令集扩展>` 的格式，如 `-mno-avx2`（禁用 AVX2）。

[gcc-x86-options]: https://gcc.gnu.org/onlinedocs/gcc-16.1.0/gcc/x86-Options.html
[gcc-aarch64-options]: https://gcc.gnu.org/onlinedocs/gcc-16.1.0/gcc/AArch64-Options.html
[gcc-loongarch64-options]: https://gcc.gnu.org/onlinedocs/gcc-16.1.0/gcc/LoongArch-Options.html

- [x86 及 x86-64][gcc-x86-options]: `-msse2`, `-msse4.2`, `-mavx`, `-mavx2`, `-mavx512f`, `-mavx10.1` 等（均为开关向量指令集扩展支持的选项）
- [ARM AArch64][gcc-aarch64-options]: AArch64 后端尚未提供 `-m<指令集扩展名>` 格式的开关。若要精细控制指令集扩展，需要传入 `-march=armv8-a+[no]指令集名` 格式的参数
- [LoongArch64][gcc-loongarch64-options]: `-mlsx` 和 `-mlasx`

发行版预设的架构微调开关一般只包括 `-march=` （或 `-mcpu=`）和 `-mtune=`。部分架构可能需要单独指定诸如 `-mfix-loongson-llsc`（绕过 MIPS 架构的龙芯 3 号上的原子操作问题）等绕过架构特定问题的参数。由于其余的参数数量繁多且大部分参数不参与发行版基线定义，因此不做详细介绍。

## 其他常用的架构微调选项

以下是发行版开发及其他场景下会观察到的架构微调选项：

| 选项名称 | 作用 | 适用的架构 | 备注 |
| :----:   | :--- | :--------: | :--- |
| `-m32`, `-m64` | 控制生成代码的位宽，即生成 32 位代码还是 64 位代码 | x86-64, PowerPC, MIPS | 这些架构理论可以原生运行低于当前处理器位宽的程序；MIPS 由 `-mips32*` 和 `-mips64*` 控制 |
| `-mbig-endian`, `-mlittle-endian` | 控制生成代码面向的处理器端序模式 | AArch64, ARM, MIPS, PowerPC, RISC-V | 大端序处理器和平台在消费领域中并不常见 |
| `-msoft-float`, `-mhard-float` | 控制生成代码采用的浮点运算方式，即硬件浮点单元、软件浮点运算实现 | AArch64, ARM, LoongArch, MIPS, PowerPC, RISC-V, x86-64 | - |

请注意，部分架构定义的参数有时并不采取上述 “通用” 选项名称，请参考目标架构的选项文档。

# 代码优化参数

现代编译器通常具有强大的优化引擎。也就是说，编译器能够在编译代码的过程中检测代码中能够检测特定模式的逻辑、循环、操作、函数调用，并将其原地替换成更加高效的版本。

现代编译器通常用 `-O<优化等级>` 参数来控制编译期间运行的优化处理过程 (Pass)：

- `-O0`: 不做优化，生成直接对应源程序的代码
- `-O1`: 运行简单的优化过程，平衡编译用时及代码性能
- `-O2`: 运行尽可能多的、更为复杂的优化过程，会延长编译时间，但能够处理大多数源程序中需要优化的模式
- `-O3`: 运行较为激进的优化过程，会显著延长编译时间，且运行的优化过程可能会破坏程序原有的逻辑

此外，GCC 还有一些其他预设：

- `-Os`: 运行 `-O2` 定义的优化过程，但除去代码对齐等会显著增加二进制大小的优化过程；同时尽可能地内联较短的函数
- `-Og`: 运行 `-O1` 定义的优化过程，但除去所有会改变代码顺序或结构的优化过程，以在尽可能提升性能的情况下不影响调试过程，保证源码与二进制一一对应
- `-Oz`: 运行比 `-Os` 更为激进的优化过程，可能会降低生成代码的性能
- `-Ofast`: 运行比 `-O3` 更为激进的优化过程，并故意违反语言标准规定的情况，如存数据竞态条件 (Data store races)

{% card(type="danger", title="切勿贪杯") %}
发行版维护方面通常规定最高允许的全局编译器优化预设是 `-O2`。`-O3` 所运行的优化过程已非常激进，会暴露出程序内部的未定义行为等缺陷，进而导致编译出的二进制程序行为不一致或根本无法启动。有一些 GCC 支持的架构后端无法完全处理 `-O3` 开启的部分优化过程，进而导致编译器内部错误 (ICE, Internal Compiler Error)。因此，在这些编译器后端支持情况较差的架构上应慎用 `-O3`。

当然，发行版内可以在完整测试通过的前提下单独为某个软件包使用 `-O3`，也有一些项目在编译时会自行设置 `-O3`。在这种情况下，除非编译器或生成的程序行为异常，否则无需强制覆盖优化等级。

## 链接时优化 (LTO)

链接时优化 (Link Time Optimization)，在 MSVC 编译器中称为链接时代码生成 (Link Time Code Generation)，是在链接二进制文件的过程中所执行的优化步骤。LTO 允许链接器将参与链接过程的所有二进制文件重新交给编译器，由编译器在整个程序范围内执行优化过程，如去除未使用的符号、合理安排二进制的布局，并且能够将所有参与链接的文件视为一整个需要编译的文件，对所有函数运行编译器启用的优化步骤。

LTO 能在一定程度上提升二进制程序的性能，但也有可能会过度优化导致程序行为异常。由于 LTO 是在最终链接过程中将组成整个程序的二进制文件全部交给编译器处理，因此 LTO 会显著增加编译时间，以及所需的内存。对 Web 浏览器等大型项目执行 LTO 时通常需要数十甚至数百 GB 的内存。

GCC 和 Clang 可以通过 `-flto` 启用 LTO。同样地，`-fno-lto` 强调编译器不要启用 LTO。

# 代码加固

C/C++ 语言本身并不提供类似于 Rust 语言的运行时边界和缓冲区溢出检测。尽管如此，glibc、libstdc++ 甚至编译器本身也提供一些代码加固功能。

## GCC 提供的加固功能

### 位置无关代码

位置无关代码能够加载在内存空间的随机地址上，而不影响其分支、跳转、静态变量引用及函数调用。因为所有分支的地址均为相对偏移量，静态变量及指针及函数调用地址需要通过全局偏移量表 (Global Offset Table) 获取，取代了链接时硬编码的绝对地址。这样做能够在一定程度上抵御 libc 跳转攻击 (Return-to-libc attack)。

GCC 定义了四种控制生成位置无关代码的开关：

- `-fpic`: 使用小代码模型生成位置无关代码（GOT 表的大小有限）
- `-fPIC`: 使用大代码模型生成位置无关代码（支持无限制的 GOT 大小）
- `-fPIE`: 指定链接器生成可重定位的二进制可执行文件

### 栈溢出保护

部分 GCC 支持的架构带有 `libssp` 支持，该库实现了栈溢出保护 (Stack Smashing Protection) 机制。SSP 通过检测存放在栈内的固定值是否被覆盖来判断栈内是否发生过缓冲区溢出等情形。这样做能够抵御覆盖返回地址的缓冲区溢出攻击。

- `-fstack-protector`: 自动在受影响的代码周围插入缓冲区溢出检查
- `-fstack-protector-strong`: 同上，但同时监测函数内定义的静态数组及函数内其他变量的引用

## 链接器提供的加固功能

链接时能够帮助加固产生的二进制的机制有：

- 只读重定位表 (Relocation Read-Only) 机制将 GOT 表所在的内存页标记为只读，防止 GOT 表中的指针被覆盖，以抵御基于缓冲区溢出的攻击
- 立即解析符号 (Bind Now) 机制在程序由动态链接器加载时立即解析程序依赖的所有外部函数（即符号），而非随程序运行动态解析，防止 PLT 表中的函数指针被覆盖

以上两种机制可以通过如下参数启用：

```
ld [其他参数] -z relro -z now
```

使用 GCC 直接生成可执行文件时，需要使用 `-Wl` 将两个参数传递到链接器：

```
gcc [其他参数] -Wl,-z,relro -Wl,-z,now
```

## glibc 头文件提供的加固功能

glibc 提供了 `_FORTIFY_SOURCE` 预处理宏，以对常见的不规范内存操作执行边界检查，如 `memcpy`, `memset`, `strcpy`, `sprintf` 等需要引入分配及释放内存操作的函数。与此同时该宏在 GCC 和 Clang 中也会引入额外编译期间的静态检查，并警告不规范的行为。`_FORTIFY_SOURCE` 是一种非常简单且性能损耗较低的加固机制，但不能检测程序运行期间的所有错误。

glibc 的加固机制分两个等级：

- `_FORTIFY_SOURCE=1`: 编译期间静态检查可能的缓冲区溢出，且允许溢出至父容器
- `_FORTIFY_SOURCE=2`: 编译期间插入运行时检查，且限制内存操作范围为该变量的大小（无法溢出至父容器）

该宏需要在编译程序时通过 `-D` 参数指定。发行版默认启用 `_FORTIFY_SOURCE=2`:

```
gcc [其他参数] -D_FORTIFY_SOURCE=2
```

# 运行时异常行为监测

异常行为监测程序 (Sanitizer) 是编译时插入或链接的、用于检测程序运行期间出现的各类常见的不规范行为及错误。目前编译器能够集成以下几种监测程序：

- [内存异常行为监测 (AddressSanitizer, ASan)][asan]: 监测程序运行期间出现的缓冲区溢出、释放后使用、重复释放及内存泄漏等情形
- [未定义行为监测 (Undefined Behavior Sanitizer, UBSan)][ubsan]: 监测程序运行期间出现的整数上溢或下溢、浮点数转换超限、非对齐或空地址访问等未定义行为
- [线程异常行为监测 (Thread Sanitizer)][tsan]: 监测程序运行期间出现的线程间共享数据的竞态条件

此类监测程序属于调试辅助功能，在调试程序的行为问题期间非常有用。因此发行版日常开发时不会启用这些监测程序。

# 链接参数

前文讲到，`gcc` 和 `clang` 属于前端驱动程序，负责调用对应语言的后端并自动完成编译、汇编及链接步骤。因此，这些前端驱动程序也提供给链接器传递参数的方式。前端驱动程序提供有直接对应的常用参数，也提供 `-Wl` 参数，用于给链接器传递自定义参数。以下是前端驱动程序提供的直接对应链接器的常用参数：

- `-l<共享库名称>`: 链接系统里的 `lib<共享库名称>.so`（动态库）或 `lib<共享库名称>.a`（静态库）
- `-L<共享库搜索路径>`: 在指定的位置上查找共享库
- `-fuse-ld=链接器名称`: 指定 GCC/Clang 使用的链接器，如 `-fuse-ld=lld`、`-fuse-ld=mold`

对于其他前端驱动程序没有提供的链接器参数，可以使用 `-Wl` 参数传递，并使用半角逗号隔开需要传递的参数。此类参数有两种格式，以下例子均相当于 `ld -X -Y -z relro`:

- `-Wl,-X,-Y,-z,relro`
- `-Wl,-X -Wl,-Y -Wl,-z -Wl,relro`

# 预处理器参数

预处理器负责处理 `#include`、`#define`、`#if` 等预处理宏。编译器提供一些用于控制预处理器行为的参数：

- `-D<预处理宏>`: 在命令行上定义预处理宏，一般此类预处理宏的目的是编译时动态指定一些常量（如项目名称），以及开启或关闭头文件及源码中定义的条件编译块
- `-U<预处理宏>`: 取消定义命令行或编译器内部定义的预处理宏（无法控制头文件内指定的宏）
- `-I<头文件路径>`: 在指定路径内搜索头文件，可以指定多个
- `-isystem <系统头文件路径>`: 指定 `#include <...>` 语句内头文件的搜索路径
- `-iquote <系统头文件路径>`: 指定 `#include "..."` 语句内头文件的搜索路径

# 编译器行为控制参数

控制编译器行为的参数主要包含下列方面：

## 语言标准控制

GCC 的大版本更新通常伴随着默认遵循的语言标准变动或标准库的头文件变动。GCC 15 更新后将默认的 C/C++ 语言标准更换为 C/C++23 之后，欠维护的项目未能及时发布针对 C/C++23 的代码改进，则会在发行版维护过程中导致众多编译问题。软件项目通常会在编译时指定使用的语言标准，以确保能在各大编译器版本上均能编译通过：

```sh
gcc -std=c17
g++ -std=c++17
```

GCC 支持的 C 语言标准分为两大类：ISO 标准和 GNU 标准。ISO 标准由国际标准化组织负责制定颁发，GNU 标准则为 ISO 标准的第三方扩展，提供额外的数据类型、额外的属性、语法糖及内联汇编等，有时也适配新语言规范的内容（如 `gnu89`）。第一个 C 标准由美国国家标准协会颁发，随后由国际标准化组织采纳并修订。因此 C89 有时也称为 ANSI C。

常见的语言标准选项有：

- `-ansi`:代码遵循最初的 C89 规范（相当于 C 语言的 `-std=c89` 和 C++ 语言的 `-std=c++98`）
- `-std=c89`, `-std=c99`, `-std=c11`, `-std=c17`, `-std=c23`: 分别对应 ANSI C、ISO C99、ISO C11、ISO C17 及 ISO C23 标准
- `-std=gnu89`, `-std=gnu99`, `-std=gnu11`, `-std=gnu17`, `-std=gnu23`: 以上 C 标准的 GNU 扩展
- `-std=c++98`, `-std=c+=03`, `-std=c++11`, `-std=c++14`, `-std=c++17`, `-std=c++20`, `-std=c++23`: 分别对应 C++ 98/03、C++ 11、C++ 14、C++ 17、C++ 20 及 C++ 23
- `-std=gnu++98`, `-std=gnu++03`, `-std=gnu++11`, `-std=gnu++14`, `-std=gnu++17`, `-std=gnu++20`, `-std=gnu++23`: 以上 C++ 标准的 GNU 扩展

## 警告控制

编译器会检测代码中出现的极小部分不规范行为并在编译期间给出警告。目前 GCC 默认将极个别不规范行为视为编译错误（自 C99 及之后的标准中此类行为均应以错误对待）：

```c
void foo () {
    foo3 ();                // 函数未预先声明
    printf ("Hello!\n");    // 函数未预先声明（存在于 <stdio.h> 头文件中）
}

foo3 (int x) {	            // 函数定义没有返回值类型，默认为 int
    int y = x + 42;
    return;                 // 返回值类型不匹配
}

int *foo4(int x) {
    int y = x + 42;
    return &x;              // 返回局部变量的地址（该行为不会按错误对待）
}
```

在发行版包含的软件项目中会启用其他警告开关，部分要求一定代码质量的项目会强制将所有警告视为错误，以期减少代码中无意产生的不规范行为。GCC 提供数百个覆盖了代码各个方面的警告参数。由于警告参数种类繁多且均能捕获到代码中常见的各类错误，因此在这里只介绍以下参数：

- [`-Wall`][gcc-wall]: 警告代码内可以轻松改正的错误
- [`-Wextra`][gcc-wextra]: 启用指定 `-Wall` 时没有启用的一些警告
- `-Werror`: 将所有警告视为错误

除上述默认视为为错误的警告外，其余警告不会中止编译过程，但可能会导致后续汇编及链接过程失败；即便能够编译出可执行文件，也可能会出现程序行为异常的情况。

## 编译器运行模式

编译器的前端程序集成了许多步骤：默认情况下运行编译器会执行编译、汇编及链接，最终生成可执行二进制或共享库。而编译器也提供了用于控制编译器运行模式的参数，方便检查每个编译阶段的输出：

- `-E`: 使用预处理器预处理源代码文件，输出预处理后的源代码
- `-S`: 将源代码编译成汇编语言源代码，不汇编成二进制克重定位文件
- `-c`: 将源代码编译成二进制可重定位文件，不执行链接步骤
- `-o <输出文件名>`: 输出到指定文件名

不指定 `-o` 参数时 GCC 的默认输出文件名如下：

- `gcc -E` 模式下产生的内容将输出到终端（标准输出）
- `gcc -S` 模式下产生的汇编源代码将输出到 `<源码文件名>.s`，如 `main.c` 编译到 `main.s`
- `gcc -c` 模式下产生的二进制文件将输出到 `<源码文件名>.o`，如 `main.c` 编译并汇编到 `main.o`
- 标准模式下产生的二进制可执行文件名为 `a.out`

# 总结

1. 工具链包含编译器、汇编器和链接器。
2. 主流的 C/C++ 编译器有 GCC 和 LLVM。
3. GCC 和 LLVM 还支持 Objective-C/C++ 和 Fortan，并且 GCC 还有 Go、D 和 Ada 语言前端。
4. 编译器中定义了众多控制编译各个方面的参数。
