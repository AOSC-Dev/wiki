在开源世界，任何项目在发版后基本只分发该版本的源码包（俗称 Tarball，以 `tar` 格式归档的源码得名）。任何感兴趣的用户需要自行将其编译并安装到自己的系统中才能使用新发版的软件。发行版统一了编译平台，并直接以软件包的形式分发编译好的二进制软件，方便用户安装和使用。本节内容将带您探索发行版打包时使用的编译器及用法，以及一些行业黑话。

# 编译器和工具链

**“编译 (Compile)”** 一般指将文本形式的源码经多次处理及转换后产生可执行的二进制、可加载的动态链接库（或称共享库, Shared Library）的过程。**编译器 (Compiler)** 就是用于执行编译过程的程序。

现代编译器分为前端（读取源码，分析并产生中间文件）和后端（将中间文件转换为目标架构的汇编指令）两部分，以此支持多种编程语言和架构。因此，现代编译器实际上可以称为**编译器套件**。

不过严格地讲，从源码到二进制文件需要经过多个步骤，这些步骤由专门的程序负责：

1. 编译器 (`gcc`) 读取源码文件，生成汇编文件
2. 汇编器 (`as`) 将编译器生成的汇编文件汇编成二进制输出文件（此时该文件还无法执行，为可重定位文件）
3. 链接器 (`ld`) 将一系列可重定位二进制组合起来，修缮函数地址及指定的动态链接库的符号信息，链接成可执行二进制或动态链接库

{% card(type="tips") %}
由于用户执行的 “编译器” 本身只是用于驱动编译器后端的程序，因此默认情况下 “编译器” 会自动完成整套步骤。
{% end %}

尽管上述是一个简化的过程，但也可以看出，从源码到二进制的可执行程序的过程中，下一个环节要使用前一个环节生成的内容，形成了一个链条。我们把编译器和用来最终生成可执行文件的程序的组合称为工具链。

此外，C 和 C++ 语言编写的程序还需链接发行版统一使用的 C 运行时库（提供 C 语言标准库实现）和 C++ 运行时库（提供 C++ 语言标准库实现）。不同 C 库的链接库文件名不同，其实现的函数及对应的符号也不同，因此发行版提供的编译器固定面向同一套 C 和 C++ 运行时库实现。主流的 C 标准库实现有 GNU glibc（为绝大多数发行版所使用）、musl（轻量级实现）和 Bionic（为 Android 系统所使用）。

一般情况下，绝大多数发行版的工具链中，编译器由 GCC 提供，汇编器及链接器由 GNU Binutils 提供，C 运行时库由 GNU glibc 提供，C++ 运行时库由 GCC 随附的 `libstdc++` 提供。

# 发行版维护相关考虑

开始做发行版之前，我们一般考虑以下方面：

1. 架构支持范围：只支持 x86-64 架构，还是维护多个架构的移植？
2. 编译器支持：GCC 和 Binutils 若不支持该架构，则该移植将无法开始（无法构建任何代码）
3. Linux 内核支持：Linux 中若不支持该架构，该移植将无法进行（没有可以运行的内核）
4. GNU glibc 支持：如果 glibc 中没有该架构支持，则无法对该架构维护发行版相关组件（缺少 C 运行时）

{% card(type="info", title="为什么选择 glibc") %}
绝大多数发行版选择 GNU glibc 的原因主要有：

- 历史原因：Linux 内核最初发布时 GNU 项目已经几近完整，只差其 Hurd 内核没有完成开发。GNU 项目于是计划利用 Linux 内核加上 GNU 已经完备的用户组件，组成完整的 GNU 操作系统生态，因此便有了 “GNU/Linux” 这个名字
- 性能：glibc 针对所有其支持的架构均有非常激进的优化（如，手动编写的、用汇编语言实现的特定函数），保证链接 glibc 的程序拥有最佳性能
- 特性完整：glibc 不仅性能出色，还拥有 GNU 自己的特性扩展，以增强特定情况下的功能，如 Unicode 支持、国际化支持等
- 用途广泛：长期以来，绝大多数 Linux 发行版均使用 GNU glibc 作为其使用的 C 运行时库
{% end %}

此外，越来越多发行版所依赖的软件项目开始迁移至更 “安全” 的 [Rust 编程语言][rust-lang]，但由于 Rust 使用的 LLVM 后端支持范围较窄，导致发行版无法为一部分 Linux 内核仍旧支持的架构维护完整特性集（如 DEC Alpha、PA-RISC 等）。

# 主流的编译器

绝大多数发行版均选择 [GCC][gcc] 作为编译整个系统时使用的编译器。有少部分软件项目推荐 [LLVM/Clang][llvm] 作为其编译器，理由是 LLVM 有更加先进的特性，如自动向量化等，因此在特定用途下性能较 GCC 编译的程序略高。

但是，市面上还有一些其他类型的编译器（如，`tcc`），但由于其为基于兴趣开发的项目，或其架构支持或语言标准支持不完整，不适合作为构建发行版时使用的编译器，故在此不做介绍。

## GNU 编译器套件 (GCC)

GNU 编译器套件 (GNU Compiler Collection, GCC)，是 GNU 项目发起并维护的编译器项目。GCC 支持 C、C++、Objective-C/C++、Go、Fortran、Ada、D 等语言，且其构建出的程序运行稳定，因此基本认为是行业标准。

GCC 套件内一般包含以下组件（部分组件列出了其程序名）：

- `gcc`、`gdc`、`f95`、`gnat`: 编译器驱动前端（Driver，本身不负责编译程序，而是根据参数及提供的文件调用对应语言的编译器）
- `cc1`: C 语言编译器
- `cc1plus`: C++ 语言编译器
- `cc1gm2`: Modula-2 语言编译器
- `f951`: Fortran 语言编译器
- `d21`: D 语言编译器
- `gnat1`: Ada 语言编译器
- `libgcc`: GCC 编译器运行时，提供了 GCC 内建函数及部分数学运算的软件实现
- `libstdc++`: GCC 随附的 C++ 语言标准库实现

GCC [支持 50 余种架构][gcc-config]（以 `gcc/config` 中的文件夹为准，不同架构的语言支持情况不同），但 GCC 一经编译，就只能支持固定的单个架构。因此若需要为其他架构编译程序，需要单独为目标架构编译 GCC 及 Binutils。

## LLVM

[LLVM][llvm] 是 LLVM Project 发起的编译器项目。起初 LLVM 全名为 “底层虚拟机 (Low Level Virtual Macine)”，但随着 LLVM 项目的演进，该全称已失去其意义。LLVM 项目包含 LLVM 通用编译器后端，以及数个编程语言前端：

- `llc`: LLVM 中间语言 (LLVM IR) 编译器
- `lld`: LLVM 项目的链接器实现
- `clang`: C/C++ 编译器前端
- `flang`: Fortran 语言编译器前端
- `libc++`: LLVM 的 C++ 标准库实现
- `compiler-rt`: LLVM 项目的编译器运行时库实现（对应 `libgcc`）

相对于 GCC，LLVM [支持的架构较少][llvm-targets]。但 LLVM 编译器套件非常灵活，编译出的 LLVM 套件可以支持多种架构。即便如此，如需为其他架构编译程序，仍需对应架构的 Binutils 和 C 标准运行库等组件。

此外，LLVM 后端的模块化实现使得其他项目能够轻松集成 LLVM，如 Rust 编译器 `rustc` 就使用 LLVM 作为其编译器后端，因此 Rust 语言的架构支持基本与 LLVM 的架构支持强绑定。

# 运行编译器

{% card(type="tips") %}
安同 OS 采用 GCC 作为标准发行版编译器。因此除非另行说明，本入门指南均以 GCC 为准。
{% end %}

编译器的命令行设计形式一般为：

```sh
编译器程序名 [其他编译器参数] -o 输出文件 输入文件 [输入文件 2 ...]
```

例如，编译并运行 Hello World 程序：

```console
$ gcc -o hello hello.c
$ ./hello
Hello, World!
```

# 编译器参数的分类

您已经注意到上面的命令行示例中可以指定 “其他编译器参数”。虽然不指定其他任何编译器参数时也能编译出程序，但编译器参数可以更精细地控制编译过程的每个方面。

GCC 套件中定义了[数百个编译器参数][gcc-options]，这些编译器参数主要分为以下几类：

- 目标架构微调：指定生成的代码至少要在哪一代指令集的处理器上运行 (`-march`)，以及在哪一代指令集的处理器上性能最佳 (`-mtune`)，以及可能需要关闭或打开的指令集扩展等
- 代码优化：控制编译过程中代码经过的优化流程，一般编译器优化预设分四个等级，以及一些特殊的优化预设
- 代码加固：自动在生成的代码中插入加固代码，如栈溢出保护，以及使用 C 标准库随附的加固功能
- 运行时异常监测：自动在生成的代码中插入或链接监测运行时各种未定义行为的代码
- 控制链接过程：调用 `gcc` 等前端驱动程序时可以向链接器传递选项，如需要链接的动态库、动态库查找位置等
- 宏定义：除了代码中的 `#define` 之外，可以在命令行中通过 `-D` 及 `-U` 参数定义或取消定义预处理宏
- 头文件控制：指定预处理期间的头文件搜索目录

接下来，我们将一一讲述上述方面中较为重要的编译器参数。

# 架构微调参数

## 发行版的基线 (Baseline)

随着处理器的演进，其使用的指令集也会随之扩展。一般来讲，新一代指令集可以向后兼容之前版本的指令集。为旧版指令集编译的程序可以在任何之后版本指令集的处理器上运行，但反之则无法运行：为新版指令集编译的程序使用了旧版指令集不支持的指令。

基线定义了发行版分发的软件最低可以运行的指令集版本。发行版的基线通常非常保守，以保证发行版中的软件可以在较广范围的处理器上运行。

### 发行版基线定义的例外情况

尽管如此，不是所有软件项目都可以运行在发行版基线定义的最低指令集版本的处理器上。其背后的原因基本可以认为只有一个：此类软件为了加快运行速度，在代码中使用了新的指令集扩展（尤其是向量指令集扩展）处理数据，而带有该指令集扩展的指令集版本比发行版基线定义的高。

典型的例子均为模拟器、专业用途或科学用途的软件，如 Blender 在 x86-64 架构上最低要求支持 SSE4.2 指令集扩展的处理器，而 SSE4.2 不是大多数发行版定义的基线（初代 x86-64）的一部分；RPCS3（PS3 模拟器）尽管支持初代 x86-64 指令集，但在支持 AVX 和更新版本的向量指令集扩展的处理器上性能更佳；Eden（Nintendo Switch 模拟器）则要求处理器至少支持 FMA 指令集扩展。

## `-march` 和 `-mtune`

发行版的基线均通过编译软件时指定 `-march` 和 `-mtune` 参数来实现：

- `-march` 定义生成的代码面向的指令集版本
- `-mtune` 定义代码调优期间面向的微架构

`-mtune` 不会生成面向目标指令集版本的代码，该参数会保持 `-march` 指定的指令集版本，而在指令调度、分支预测、缓存对齐等方面为指定的微架构优化，使编译的程序在 `-mtune` 指定的微架构处理器上性能最佳，但依旧可以在其他版本指令集的处理器上运行。

以下列举几个例子：

- `-march=x86-64 -mtune=arrowlake`: 生成面向初代 x86-64 指令集架构的代码，但面向 Arrow Lake 在指令层面优化生成的代码
- `-march=arrowlake -mtune=arrowlake`: 生成面向 Arrow Lake 微架构支持的指令集的代码，同时面向 Arrow Lake 微架构调优；生成的程序会尽量使用 Arrow Lake 支持的指令集中的指令（例如 AVX2）
- `-march=armv9-a -mtune=cortex-a710`: 生成面向 ARMv9.0-A 指令集的代码，面向基于 Cortex-A710 微架构的处理器调优（更早的核心不支持 ARMv9.0-A 指令集）

不过，在 Intel 处理器家族中有一处例外：

- AVX-512 指令集扩展只在一部分家族的处理器中支持，Intel 在其 12 代 Core 处理器中移除了该指令集扩展的支持。因此如果面向这些处理器家族编译程序，这些程序将无法在 12 代及之后的 Core 处理器上运行。
